## Problem Statement

产品需求通常来自零散文字、Markdown、业务文档或界面截图，内容中容易混入假设、遗漏依赖、缺少异常场景，也缺乏从需求澄清到技术交付的稳定过程。直接让大模型一次性生成 PRD 或技术方案，会产生以下问题：

- 模糊需求未经澄清便被当作事实写入文档。
- 需求结构、数据来源、交互、操作影响和依赖关系没有完整性门禁。
- PRD 作者同时承担审查职责，缺少相对独立的质量检查。
- PRD 未经人工确认便继续生成 SDD，过程不可控。
- 修改、重试、回退和重新生成缺少明确的状态约束与审计记录。
- 文档、需求快照、问题、审批和执行任务缺少统一持久化，难以追踪历史版本。
- 不同 LLM Provider、模型和本地模型的配置分散，难以按项目稳定复用。

## Solution

PRD Agent 提供一个本地单管理员的需求工作台，通过 CrewAI 多角色工作流、Python 状态机和人工门禁，把初始需求逐步推进为结构化需求、逻辑校验结果、已终审 PRD 和可执行 SDD。

系统支持文本与多格式附件输入；由异步 Worker 调用不同职责的 Agent；由服务端代码决定阶段是否可推进，而不是允许 LLM 直接改变状态。用户可以在 Web 管理台或 CLI 中查看项目状态、回答问题、提交反馈、确认阶段、回退修改、查看版本化产物，并管理项目绑定的 LLM 配置。

## 产品目标

1. 将模糊输入转化为可追溯、可确认的结构化需求。
2. 在生成 PRD 前完成需求完整性检查和七维逻辑校验。
3. 通过独立 PRD 终审和受控修订降低文档逻辑遗漏。
4. 仅在用户确认后，从终审通过的 PRD 生成 SDD。
5. 保留需求、产物、审批和工作流事件的版本与审计信息。
6. 让耗时的 LLM 任务异步执行，并防止重复提交和并发冲突。
7. 提供可在本地安装、配置、登录和使用的完整 Web 工作台，同时保留 CLI 操作入口。

## 非目标

- 不直接生成或修改业务代码。
- 不替代项目管理、研发排期或发布管理系统。
- 不让 LLM 自主跳过人工确认或改变工作流阶段。
- 不把 SDD 生成等同于技术专家签字或研发交付闭环。
- 不在当前版本提供多用户、组织、角色权限或云端协作。

## 目标用户与角色

| 角色 | 核心诉求 |
| --- | --- |
| 产品负责人 | 从模糊想法得到结构完整、经过审查的 PRD |
| 需求分析人员 | 系统化识别需求缺口、依赖、边界和异常场景 |
| 技术负责人/开发人员 | 从已确认 PRD 获得结构化 SDD 和明确的技术契约 |
| 本地系统管理员 | 完成数据库、管理员账号、LLM 配置和运行环境设置 |
| 自动化 Worker | 串行、安全地执行长耗时工作流任务并保存结果 |

## 已实现的端到端业务流程

```text
初始需求（文本/Markdown/附件）
  ↓
STRUCTURING 需求结构化
  ↓ 完整性门禁 + 用户回答
LOGIC_VALIDATING 七维逻辑校验
  ↓ 阻断/重要问题处理 + 用户确认
PRD_TYPE_CONFIRMING 产品类型识别与确认
  ↓ 用户确认
PRD_GENERATING PRD 生成
  ↓
PRD_REVIEWING 独立终审
  ├─ 未通过 → PRD_REVISING → 再次终审
  ├─ 未通过 → 受控回退/强行同意（记录理由）
  └─ 通过
       ↓
SDD_CONFIRMING SDD 生成前确认
  ↓ 用户确认，或回退修改
SDD_GENERATING SDD 生成
  ↓
COMPLETED 完成
  └─ 可重新生成新版 SDD
```

阶段状态包括 `running`、`waiting_user`、`failed`、`completed`。所有合法阶段跳转由服务端状态机约束，非法跳转必须被拒绝。

## 功能需求

### FR-01 首次运行与数据库设置

1. 当系统尚未配置数据库时，应进入安装模式，只开放健康检查和数据库设置相关接口。
2. 用户应能填写正式库和测试库的 MySQL 连接地址并执行连通性测试。
3. 数据库连接必须使用 `mysql+pymysql` 且包含 `charset=utf8mb4`。
4. 测试数据库名称必须以 `_test` 结尾，且不得与正式数据库指向同一数据库。
5. 保存配置时应保留 `.env` 中无关配置，只更新数据库连接项。
6. 系统应能通过 Alembic 将数据库升级到当前最新结构。

### FR-02 单管理员认证

1. 首次使用时，用户应能创建唯一管理员账号。
2. 管理员密码必须使用 Argon2 哈希保存。
3. 登录成功后，系统应使用 HttpOnly 会话 Cookie 维持登录状态。
4. 会话在服务端只保存令牌哈希，不保存原始令牌。
5. 过期或无效会话必须被拒绝。
6. 除安装、认证和健康检查接口外，业务 API 必须要求已登录。
7. 管理员应能主动退出并使当前会话失效。

### FR-03 LLM 配置管理

1. 管理员应能新增、编辑、查看、归档和恢复 LLM 配置。
2. 配置内容应包括名称、Provider、模型、API Key、Base URL、Temperature、超时时间和原生 Structured Output 策略。
3. 系统应支持 OpenAI、DeepSeek、Ollama 和 OpenAI-compatible 自定义端点等 Provider 形态。
4. 本地模型可在 Provider 允许时不填写 API Key。
5. API Key 返回前必须脱敏，不得通过查询接口返回原值。
6. 编辑时 API Key 留空应保留旧值；只有明确选择清除时才删除旧值。
7. 用户应能指定全局默认配置，并可为单个项目切换配置。
8. 用户应能发起异步连通性测试并查看测试任务状态。
9. 工作流任务入队时应记录配置 ID 和版本；若排队后配置发生变化，Worker 应拒绝使用漂移后的配置执行任务。

### FR-04 项目创建与需求输入

1. 用户应能仅输入需求文字、仅上传附件，或同时使用两种方式创建项目。
2. 文字输入应支持 Markdown 编辑与安全预览；预览不得执行原始 HTML。
3. 项目创建时应能选择 LLM 配置；未指定时使用全局默认配置。
4. 支持的附件格式应包括：
   - 文本：`.md`、`.txt`
   - 网页：`.html`、`.htm`
   - 文档：`.pdf`、`.docx`
   - 图片：`.png`、`.jpg`、`.jpeg`、`.webp`
5. 单项目默认最多上传 8 个附件，附件总大小默认不超过 20 MB。
6. 系统应保存原始附件，并记录原始文件名、存储路径、MIME 类型、大小、SHA-256、类型和处理状态。
7. 文件名和存储路径必须经过安全处理，禁止目录穿越。
8. 文本、HTML、PDF 和 DOCX 应提取为文本需求来源；HTML 提取时应忽略脚本和样式内容。
9. 图片应使用项目所选 LLM 的 Vision 能力提取需求事实；非多模态模型必须明确失败，不得静默降级。
10. 附件处理结果应只追加一次到需求来源，避免 Worker 重试导致重复输入。
11. 用户应能在项目工作台查看附件处理状态、错误信息并下载原始附件。

### FR-05 需求结构化

1. 系统应将全部原始输入整合成完整需求规格，而不是只返回本轮增量。
2. 结构化需求至少应覆盖以下五个完整性维度：
   - 模块与功能
   - 数据来源
   - 交互逻辑
   - 操作影响
   - 依赖关系
3. 需求规格还应支持状态、业务规则、异常、假设和原始输入追踪。
4. 未经用户确认的内容必须进入假设，不得伪装为事实。
5. 缺失信息必须形成稳定编号的澄清问题，编号格式为 `Q-XXX`。
6. 用户应能逐条回答所有必答问题，也可提交补充反馈。
7. 未完成五维完整性检查或仍有开放问题时，系统不得进入逻辑校验阶段。
8. LLM 返回内容不得覆盖或伪造由工作流维护的原始需求来源。

### FR-06 七维需求逻辑校验

1. 系统应对完整需求执行以下七个维度的扫描：
   - 依赖关系
   - 状态流转
   - 边界条件
   - 操作影响
   - 逆向操作
   - 交互一致性
   - 异常场景
2. 逻辑问题必须使用稳定编号 `L-XXX`，并标记为 `blocking`、`important` 或 `suggestion`。
3. 用户回答后，系统应重新扫描并记录已解决问题，不得简单删除历史问题。
4. 开放的阻断问题和重要问题必须阻止进入下一阶段。
5. 建议类问题可以保留，不阻塞流程。
6. 重要问题允许用户填写原因后豁免；阻断问题不得豁免。
7. 问题被解决或豁免时必须保存解决方案或豁免原因。

### FR-07 产品类型识别与确认

1. 系统应识别一个主产品类型，并可识别多个次产品类型。
2. 支持的产品类型包括：管理后台、C 端应用、API/服务、数据产品、中台/平台、硬件/嵌入式。
3. 识别结果应包含匹配特征和判断理由。
4. 产品类型必须由用户确认后才能生成 PRD。
5. 用户提交修正意见后，系统应重新识别，不得自动视为已确认。

### FR-08 PRD 生成

1. 系统应由 PRD 撰写角色基于已确认的需求规格和产品类型生成中文 PRD。
2. PRD 必须包含九个规定章节：文档信息、修订历史、项目背景与目标、核心名词定义、功能需求总览、详细功能需求、非功能需求、全局交互规范、附录。
3. 每个功能应包含字段规格、交互步骤、唯一业务规则以及 Loading、Empty、Error、Data 页面状态。
4. PRD 中的业务规则编号必须符合 `R-XXX` 且保持唯一。
5. 生成内容不得添加输入需求中不存在的业务能力；不确定项必须标记待确认。
6. 系统应对输出 JSON、标识符和必需章节执行代码级 Guardrail 校验。
7. 每次生成或修订的 PRD 都必须保存为新的版本化产物。

### FR-09 PRD 独立终审与修订

1. PRD 终审必须由与 PRD 作者职责隔离的审查角色执行。
2. 终审必须且只能覆盖：逻辑准确性、流程闭环性、场景覆盖度、交互统一性、用户使用逻辑。
3. 任一维度失败或存在阻断问题时，终审不得判定通过。
4. 终审报告应作为独立版本化产物保存，不得静默改写被审 PRD。
5. 终审未通过时，系统应支持根据审查意见生成新版 PRD，并再次进入终审。
6. 用户应能提交额外反馈后触发修订。
7. 自动修订轮数应受配置限制，默认最多 3 轮。
8. 终审失败时，用户应能受控回退到逻辑校验或 PRD 修订等合法目标。
9. 用户可强行同意终审，但必须填写理由，系统必须记录审批和审计事件。

### FR-10 SDD 确认、生成与重生成

1. 只有终审通过或被有理由地强行同意后，项目才能进入 SDD 确认。
2. 用户必须显式确认后才能生成 SDD。
3. 在 SDD 确认阶段，用户应能回退到 PRD 修订或产品类型确认。
4. SDD 应由解决方案架构角色根据终审后的 PRD 生成，不得增加 PRD 中不存在的业务能力。
5. 每个 SDD 模块必须包含：功能契约、页面结构与组件树、数据模型定义、API 接口定义、数据库表设计、核心交互流程、测试验收标准、技术决策与约束。
6. SDD 必须保存为版本化产物，并记录技术默认值。
7. 项目完成且已有 SDD 后，用户应能重新生成一个新版本 SDD。
8. SDD 重生成不得覆盖旧版本。

### FR-11 受控回退、重试和失败恢复

1. 工作流只允许回退到当前阶段声明的合法目标。
2. PRD 终审阶段可按规则回退到逻辑校验或 PRD 修订。
3. SDD 确认阶段可按规则回退到 PRD 修订或产品类型确认。
4. 非法回退、非法推进和归档项目上的工作流写操作必须被拒绝。
5. LLM 或附件处理失败时，项目应显示失败状态和错误信息。
6. 失败项目应提供重试入口；PRD 终审失败时还应提供反馈、回退和强行同意入口。
7. 门禁未通过属于可预期业务结果，Worker 应返回门禁错误而不是把任务错误地标记为系统崩溃。

### FR-12 项目工作台

1. 用户应能查看项目列表、阶段、阶段状态、轮次、标题和摘要。
2. 用户应能按标题或项目 ID 搜索。
3. 用户应能在进行中项目和已归档项目之间切换。
4. 项目可归档和恢复；归档后不允许继续执行工作流动作。
5. 项目详情应展示九阶段进度、需求摘要、开放问题、逻辑问题、门禁错误、附件和后台任务状态。
6. 页面只应展示当前阶段合法且可用的操作。
7. 存在运行中任务时，应禁用冲突操作。
8. 用户应能为当前项目切换 LLM 配置。

### FR-13 产物查看与导出

1. 用户应能查看项目全部 PRD、PRD Review 和 SDD 版本列表。
2. 用户应能按产物类型和版本打开 Markdown 内容。
3. Web 端 Markdown 渲染必须经过 DOMPurify 清理。
4. 用户应能下载指定版本的 Markdown 产物。
5. CLI 应支持将最新 PRD 或 SDD 导出到指定路径或默认输出目录。
6. 旧版本必须保留，后续生成不得覆盖历史产物。

### FR-14 异步任务与幂等控制

1. 创建项目、推进工作流、提交回答、回退、重生成和 LLM 测试等长耗时操作应通过 MySQL 任务队列异步执行。
2. 写请求应使用幂等键，重复请求不得重复创建项目、任务或写入结果。
3. 同一项目同一时间只能存在一个活动工作流任务。
4. 多个 Worker 不得重复领取同一任务。
5. 任务应记录排队、运行、成功、失败状态及开始、结束时间和错误信息。
6. 项目使用的 LLM 配置版本必须随任务固定，防止执行期间配置漂移。

### FR-15 持久化、版本与审计

1. 系统应使用 MySQL 8.0+ 和 utf8mb4 持久化业务数据。
2. 应持久化项目状态、需求快照、澄清问题、逻辑问题、产物、审批、工作流事件、LLM 配置、任务、管理员和会话。
3. 需求快照和产物版本号在并发写入时必须串行分配，避免重复版本。
4. 项目删除时，其需求、问题、产物、任务和附件元数据应按外键规则级联清理。
5. 审批和关键工作流动作应形成可追踪事件。
6. 原始附件文件由共享上传目录持久化，API 与 Worker 必须挂载同一目录。

### FR-16 Web 与 CLI 双入口

1. Web 应提供数据库设置、管理员设置/登录、项目列表、新建项目、项目工作台、产物查看和 LLM 配置页面。
2. 前端路由应在数据库未配置时跳转安装页，在未登录时跳转登录页。
3. CLI 应提供 `new`、`resume`、`status`、`export`、`db-upgrade`、`api` 和 `worker` 命令。
4. CLI 创建、恢复和查看项目时，应显示当前阶段、问题、逻辑问题、产品类型或完成状态。
5. Web API 默认只允许监听 localhost，以符合当前本地工作台定位。

## User Stories

1. As a 产品负责人, I want 输入一段模糊需求, so that 系统可以开始结构化分析。
2. As a 产品负责人, I want 使用 Markdown 编写并预览需求, so that 我可以保留清晰的层级和表格表达。
3. As a 产品负责人, I want 上传已有的需求文档和截图, so that 不必手工复制所有信息。
4. As a 产品负责人, I want 同时提交文字和附件, so that 补充说明与原始材料可以一起被分析。
5. As a 产品负责人, I want 查看并下载原始附件, so that 可以核对需求来源。
6. As a 产品负责人, I want 回答稳定编号的澄清问题, so that 需求缺口能被逐项关闭。
7. As a 产品负责人, I want 看到结构化完整性门禁, so that 不完整需求不会过早进入 PRD。
8. As a 需求分析人员, I want 查看七维逻辑扫描结果, so that 能发现依赖、状态、边界和异常问题。
9. As a 需求分析人员, I want 逐条解决或有理由地豁免非阻断问题, so that 每项决策都有记录。
10. As a 产品负责人, I want 确认或修正产品类型, so that PRD 内容能匹配管理后台、C 端、API、数据、平台或硬件场景。
11. As a 产品负责人, I want 获得章节完整的中文 PRD, so that 文档可以进入评审和实施准备。

## Implementation Decisions

- 使用 CrewAI Flow 组织工作流，但阶段、门禁、回退和审批由 Python 代码控制。
- 将产品分析、PRD 撰写、PRD 独立终审和 SDD 架构设计拆分为不同 Agent 职责；Agent 禁止委派并限制最大迭代次数。
- 使用 Pydantic 模型约束需求、问题、逻辑问题、产品类型、PRD Review 和 SDD 结构。
- 对 LLM 输出执行标识符唯一性、规定章节和审查维度 Guardrail；对不适合原生 Structured Output 的 Provider 使用提示词约束 JSON。
- 使用 FastAPI 提供版本化 API，统一返回成功/错误包装，并通过异常处理映射业务错误。
- 使用 Vue 3、TypeScript、Element Plus、Vue Router 和 TanStack Vue Query 构建本地管理台。
- 使用 markdown-it 渲染 Markdown，并使用 DOMPurify 清理 HTML。
- 使用 MySQL 作为唯一业务数据库；不支持 SQLite 等替代后端。
- 使用 SQLAlchemy Repository 隔离持久化，使用 Alembic 管理结构升级。
- 使用 MySQL 表实现持久化异步队列、幂等写和项目级任务互斥。
- 需求快照、PRD、PRD Review 和 SDD 采用线性版本号；该版本号表示文档修订，不代表产品 MVP/V1/V2 路线图。
- 原始附件保存在文件系统，元数据、哈希、提取文本和状态保存在 MySQL。
- 图片复用项目 LLM 配置进行 Vision 分析，不单独增加图片模型配置。
- 当前认证模型为本地单管理员；密码使用 Argon2，会话使用 HttpOnly Cookie 和服务端令牌哈希。
- 当前 API 约束为 localhost 工作台，符合本机单用户产品定位。
- LLM 配置中的 API Key 在当前本机模式下明文保存在 MySQL，但读取接口只返回掩码。

## Testing Decisions

### 测试原则

- 优先测试用户可观察行为和最高层业务接缝，不依赖内部实现细节。
- 工作流以完整状态推进、门禁拒绝、回退、修订和版本产物为主要断言。
- API 以请求契约、认证、幂等、合法动作和返回状态为主要断言。
- Repository 以 MySQL 的事务、唯一约束、并发版本和任务领取语义为主要断言。
- 前端以路由守卫、请求格式、用户提交和阶段展示为主要断言。

## 数据与状态规则

### 核心数据对象

| 对象 | 用途 |
| --- | --- |
| Project | 当前阶段、阶段状态、轮次、完整工作流状态和项目配置绑定 |
| Requirement Snapshot | 每轮结构化需求的历史快照 |
| Question | 澄清问题、答案和关闭状态 |
| Logic Issue | 逻辑问题、严重级别、解决方案或豁免原因 |
| Artifact | PRD、PRD Review、SDD 的版本化 Markdown |
| Approval | 产品类型确认、终审覆盖、SDD 确认等审批记录 |
| Workflow Event | 阶段和关键动作审计事件 |
| Workflow Job | 异步任务、幂等键、执行状态、配置版本和错误 |
| LLM Config | Provider、模型、参数、密钥、默认和归档状态 |
| Project Attachment | 原文件元数据、哈希、类型、提取文本和处理状态 |
| Admin User/Session | 单管理员账号和服务端会话 |

### 关键业务规则

- BR-001：LLM 输出不能直接改变工作流阶段。
- BR-002：结构化需求未满足五维完整性或仍有开放问题时不得推进。
- BR-003：开放的 blocking 或 important 逻辑问题不得推进。
- BR-004：blocking 问题不得豁免。
- BR-005：产品类型未经用户确认不得生成 PRD。
- BR-006：PRD 终审未通过时不得直接进入 SDD 确认，除非用户填写理由执行强行同意。
- BR-007：SDD 必须在用户显式确认后生成。
- BR-008：任何新 PRD、Review 或 SDD 都必须新增版本，不得覆盖历史版本。
- BR-009：归档项目不得执行工作流写操作。
- BR-010：同一项目同时只能存在一个活动工作流任务。
- BR-011：重复幂等键必须返回已有结果，不得重复执行。
- BR-012：图片附件只能由支持多模态的模型处理。

## 非功能需求

### 安全

- 业务 API 必须经过管理员会话认证。
- 密码不得明文存储，API Key 不得通过读取 API 返回原值。
- Markdown 渲染必须禁用原始 HTML 并进行 DOM 清理。
- 附件路径必须限制在上传根目录内。
- 数据库连接必须使用独立正式库和测试库。

### 一致性与可靠性

- 状态转换、版本分配、任务领取和关键写入必须在数据库事务中完成。
- 重复请求、重复 Worker 领取和并发产物生成不得产生重复记录。
- API 与 Worker 必须共享附件卷，避免任务无法读取原文件。

### 可用性

- 所有异步操作都应展示可理解的任务状态和失败信息。
- 前端应提供加载、空数据、错误和重试状态。
- 当前阶段不可用的按钮不得展示或必须禁用。

### 可维护性

- 工作流、门禁、Agent、Task、Repository、API 和 Web 展示保持职责分离。
- 数据库结构变化必须提供 Alembic 迁移。
- 业务状态和值对象使用显式枚举和结构化模型。

## Out of Scope

以下能力不属于当前“已完结功能”，不得从本文档推断为已经交付：

1. PRD 生成前的独立技术栈/技术约束录入阶段。
2. 独立的技术专家可行性评审门禁。
3. SDD 之后覆盖开发、测试、验收和上线结果的技术闭环。
4. 产品路线图、MVP/V1/V2 能力拆分、发布计划或跨版本依赖规划。
5. 多用户、团队、组织、RBAC、SSO 和租户隔离。
6. 在线协作评论、消息通知、审批流集成和外部 Issue/项目管理系统集成。
7. 自动生成代码、创建 Git 分支、提交代码或部署业务应用。
8. API Key 加密存储、KMS/Secret Manager 集成和面向公网的完整安全加固。
9. 对扫描版 PDF 的 OCR；当前 PDF 能力只提取文件中已有文本层。
10. 附件病毒扫描、内容安全审核和对象存储生命周期管理。
11. PRD/SDD 的 DOCX、PDF 等富格式导出；当前产物以 Markdown 为主。
12. 已验收的 Docker Compose 一键部署和公网生产部署。仓库已有 Compose/Nginx 配置资产，但当前 API 监听约束、CLI 启动参数和健康检查路径尚未完成一致性验证，故不计入已完结功能。

## Further Notes

- 当前产品定位是“本地单用户需求工作台”，不是 SaaS 协作平台。
- 当前产物版本是文档修订历史，不等同于产品版本规划。
- PRD 独立终审属于产品文档质量检查；现有 SDD 生成属于技术方案产出，两者都不能替代尚未实现的技术专家评审与技术闭环。
- API Key 明文存储仅适用于已声明的本机单用户模式。若部署到远程服务器，必须先完成密钥加密、网络边界和访问控制加固。
- 本文档应在工作流阶段、数据库迁移、用户入口或安全模型发生变化时同步更新。
