| 项目 | 内容 |
| --- | --- |

| 规划版本 | v0.2 |
| 版本主题 | 项目版本规划与迭代工作流 |
| 文档类型 | 新版本规划 PRD |
| 文档状态 | 待评审 |

## 版本定位

v0.2 将 PRD Agent 从“一个项目只能执行一次需求工作流”升级为“一个项目可以规划并推进多个产品版本”。

新版本采用以下核心结构：

```text
Project（长期产品项目）
├── Version MVP
│   └── 独立需求工作流 → PRD → Review → SDD
├── Version V1.0
│   └── 继承 MVP 基线 + 增量需求 → 独立工作流
└── Version V1.1
    └── 继承 V1.0 基线 + 增量需求 → 独立工作流
```

这里的 `Version` 表示产品迭代版本；PRD v1、PRD v2 等仍表示同一产品版本内部的文档修订，两者必须在模型、界面和文案上严格区分。

## Problem Statement

当前 PRD Agent 将 `Project` 同时用作产品容器和工作流实例。一个项目创建后，只能沿固定流程生成一组需求规格、PRD、终审报告和 SDD。

当产品需要从 MVP 继续迭代到 V1、V2 时，用户只能新建互不关联的项目，或者在原项目中覆盖式地补充需求。这会造成以下问题：

- 无法表达一个长期产品包含多个迭代版本。
- 无法规划每个版本的目标、范围、优先级和前后依赖。
- 新版本无法明确继承上一版本已经确认的需求基线。
- 无法区分“新增、修改、删除、保持不变”的需求差异。
- 每个版本生成的 PRD、SDD 和文档修订记录缺少统一归属。
- 用户无法查看产品从 MVP 到后续版本的演进路线。
- 新版本需求容易重复描述既有能力，或无意破坏上一版本的业务规则。
- 当前 `Artifact.version` 和需求快照版本只能表示文档修订，不能承担产品版本语义。

## Solution

系统新增独立的产品版本层，将长期存在的 `Project` 作为产品容器，将现有九阶段工作流下沉到 `Project Version`。

用户可以：

1. 创建长期产品项目。
2. 手工规划或由 AI 辅助拆分 MVP、V1、V2 等产品版本。
3. 为每个版本定义目标、范围、非目标、优先级、依赖和验收结果。
4. 从指定基线版本创建下一版本，并继承已确认的需求规格。
5. 录入新版本的增量需求，由系统识别新增、修改、删除和不变项。
6. 确认版本差异后，为该版本启动独立的需求到 PRD/SDD 工作流。
7. 在项目维度查看版本路线、版本状态、版本差异和每个版本的全部文档。

## 产品目标

1. 支持一个产品项目管理多个有序产品版本。
2. 支持从上一版本继承需求基线，减少重复输入。
3. 支持版本范围规划和 AI 辅助拆分。
4. 支持版本间需求差异的结构化识别与人工确认。
5. 每个产品版本拥有独立、完整、可回退的九阶段工作流。
6. 保证产品版本号与文档修订号概念清晰、数据隔离。
7. 平滑迁移 v0.1 已有项目及全部历史产物。

## 成功指标

| 指标 | 目标 |
| --- | --- |
| 旧项目迁移完整率 | 100%，原工作流状态、附件和产物均可访问 |
| 新版本创建成功率 | 合法输入下 100% |
| 版本基线继承准确率 | 已确认需求对象完整继承，无原记录覆盖 |
| 版本差异可追踪率 | 所有新增、修改、删除项都有来源和确认记录 |
| 数据隔离 | 不同产品版本的问题、任务、附件和产物不串联 |
| 概念清晰度 | 界面中产品版本与文档修订版本使用不同名称和展示方式 |

## 核心概念

| 概念 | 定义 |
| --- | --- |
| Project | 长期产品项目，是多个产品版本的容器 |
| Project Version | 一次有明确目标和范围的产品迭代，如 MVP、V1.0 |
| Version Plan | 项目版本路线及各版本的目标、范围、顺序和依赖 |
| Baseline Version | 当前版本继承需求的来源版本 |
| Baseline Requirement | 基线版本最终确认的结构化需求 |
| Version Change Set | 当前版本相对基线的新增、修改、删除和不变项 |
| Workflow | 某个产品版本内部独立运行的九阶段需求工作流 |
| Product Version Code | 用户可见的版本标识，如 MVP、V1.0、2026-Q3 |
| Artifact Revision | 某产品版本内部 PRD/Review/SDD 的文档修订号 |

| 需求分析人员 | 检查版本拆分、继承关系和增量需求完整性 |
| 技术负责人 | 查看版本依赖、技术影响和每个版本的 SDD |

| 版本规划 Agent | 辅助拆分版本、识别版本目标和范围 |
| 需求工作流 Agent | 在单个产品版本内执行现有九阶段工作流 |

## 目标流程

### 新项目流程

```text
创建 Project
  ↓
输入产品愿景、业务目标、候选需求
  ↓
生成或手工创建 Version Plan
  ↓
用户确认版本路线
  ↓
选择首个版本
  ↓
确认版本目标与范围
  ↓
启动该版本九阶段工作流
  ↓
完成 PRD / Review / SDD
  ↓
创建下一产品版本
```

### 迭代版本流程

```text
选择基线版本
  ↓
创建新产品版本
  ↓
复制基线版本最终确认的需求
  ↓
输入增量需求与变更原因
  ↓
系统生成 Version Change Set
  ↓
用户逐项确认新增/修改/删除
  ↓
形成当前版本完整需求输入
  ↓
启动独立九阶段工作流
```

## 版本状态模型

产品版本状态独立于工作流阶段：

```text
PLANNED → READY → IN_PROGRESS → COMPLETED → ARCHIVED
   ↓        ↓
CANCELLED ←┘
```

| 状态 | 含义 |
| --- | --- |
| PLANNED | 已进入版本路线，但范围尚未确认 |
| READY | 版本目标和范围已确认，可以启动需求工作流 |
| IN_PROGRESS | 版本工作流已启动 |
| COMPLETED | 该版本工作流已经完成并生成 SDD |
| CANCELLED | 版本被取消，不再推进 |
| ARCHIVED | 已完成或已取消版本被归档 |

状态规则：

- `READY` 之前必须确认版本目标、范围和基线。
- 工作流首次启动时自动进入 `IN_PROGRESS`。
- 工作流进入 `COMPLETED` 时，产品版本自动进入 `COMPLETED`。
- `IN_PROGRESS` 版本不得直接删除。
- 已被其他版本作为基线的版本不得删除。
- 归档只影响默认展示，不删除任何版本数据。

## 功能需求

### FR-01 项目与产品版本分层

1. `Project` 必须从工作流实例升级为长期产品容器。
2. 一个项目必须能够包含一个或多个产品版本。
3. 每个产品版本必须属于且只能属于一个项目。
4. 每个产品版本必须拥有独立工作流状态。
5. 同一项目内产品版本代码必须唯一。
6. 产品版本必须包含内部唯一 ID，业务逻辑不得依赖可修改的展示代码。
7. 项目归档时，其版本默认不可继续操作；恢复项目后恢复可操作状态。
8. 产品版本可以独立归档，不影响同项目其他版本。

### FR-02 项目级产品信息

1. 创建项目时应录入项目名称、产品愿景和总体业务目标。
2. 项目可记录目标用户、核心问题、长期边界和全局约束。
3. 项目级信息是各版本共同背景，不应在每个版本重复维护。
4. 修改项目级信息不得静默修改已完成版本的历史需求和文档。
5. 新版本启动时应记录当时使用的项目级信息快照。

### FR-03 版本路线规划

1. 用户应能为项目创建版本路线。
2. 版本路线必须支持手工新增、编辑、排序、取消和归档版本。
3. 每个计划版本应包含：
   - 版本代码
   - 版本名称
   - 版本目标
   - 目标用户或场景
   - 计划范围
   - 明确非目标
   - 优先级
   - 前置版本
   - 计划备注
4. 版本路线应允许 MVP、语义化版本、日期版本等不同命名方式。
5. 系统不应强制用户采用固定版本命名规则，但版本代码不得为空。
6. 版本排序由显式顺序字段控制，不根据版本代码字符串自动推断。
7. 调整计划版本顺序时，应校验基线和依赖关系没有形成环。

### FR-04 AI 辅助版本拆分

1. 用户应能提交产品愿景、完整需求池和拆分偏好，让 Agent 生成版本规划建议。
2. Agent 应至少输出：
   - 建议版本列表
   - 每个版本的目标
   - 纳入范围
   - 延后范围
   - 拆分理由
   - 版本依赖
   - 主要风险和待确认问题
3. Agent 应优先保证版本形成可验证的用户价值闭环，不能只按技术模块横向切分。
4. Agent 不得将建议直接写入正式版本路线。
5. 用户必须确认后，建议才能转为计划版本。
6. 用户可以仅接受部分建议、调整顺序或修改版本内容。
7. 原始 AI 建议和用户最终确认结果都应保留审计记录。

### FR-05 版本范围管理

1. 每个版本必须维护 `In Scope` 和 `Out of Scope`。
2. 范围项应支持模块、功能、业务规则、数据、交互或非功能需求等类型。
3. 每个范围项应有稳定标识，不应只依赖自然语言标题。
4. 范围项应记录来源：项目需求池、基线继承、用户新增或 Agent 建议。
5. 同一范围项只能归属一个首发版本，但可以在后续版本中继续修改。
6. 用户应能将范围项从一个尚未启动的计划版本移动到另一个计划版本。
7. 已进入工作流的范围变更必须进入正式变更流程，不能直接拖动覆盖。

### FR-06 版本基线与继承

1. 创建非首个版本时，用户应选择一个基线版本。
2. 默认基线应为同项目最近一个 `COMPLETED` 版本。
3. 基线版本必须属于同一项目。
4. 系统应继承基线版本最终确认的完整需求规格，而不是继承其最初输入。
5. 继承内容应包括模块、功能、数据来源、交互、操作影响、依赖、状态、业务规则、异常和已确认假设。
6. 新版本继承后必须生成独立副本，不得修改基线版本记录。
7. 基线版本的 PRD 和 SDD 只作为参考，不直接复制成新版本正式产物。
8. 首个版本没有基线时，应以项目愿景和本版本原始输入建立需求。

### FR-07 增量需求输入

1. 新版本应支持 Markdown 文字和现有多格式附件输入。
2. 输入必须明确标记为当前产品版本的增量需求来源。
3. 用户应能说明本次迭代的业务原因、期望结果和约束。
4. 附件必须归属于具体产品版本，不能只挂在项目容器。
5. 用户应能引用项目级共享材料，但系统必须记录引用关系。
6. 同一附件不得因为任务重试被重复追加到版本输入。

### FR-08 版本差异识别

1. 系统应将基线需求与增量输入合并分析，生成结构化 `Version Change Set`。
2. 每个差异项必须标记为：
   - `ADDED`
   - `MODIFIED`
   - `REMOVED`
   - `UNCHANGED`
3. 修改项必须同时展示基线内容、建议新内容和变更原因。
4. 删除项必须说明删除影响，包括依赖、数据、状态、交互和兼容性影响。
5. 新增项必须说明归属模块和对既有能力的影响。
6. 系统应识别疑似重复、冲突和无法匹配基线的需求。
7. Agent 只能提出差异建议，不能自动确认删除或破坏性修改。
8. 用户必须逐项确认 `MODIFIED` 和 `REMOVED`。
9. 未确认的阻断差异不得启动正式需求工作流。

### FR-09 版本范围确认门禁

1. 产品版本从 `PLANNED` 进入 `READY` 前必须通过范围确认门禁。
2. 门禁至少校验：
   - 版本目标非空
   - In Scope 已定义
   - Out of Scope 已定义
   - 基线关系有效
   - 所有修改和删除项已确认
   - 不存在循环版本依赖
   - 不存在同一范围项在多个未启动版本重复首发
3. 门禁错误必须向用户逐项展示。
4. 用户不能通过普通“下一步”绕过门禁。
5. 当前版本范围确认应形成审批记录和审计事件。

### FR-10 版本工作流实例

1. 每个产品版本应运行现有九阶段工作流。
2. 工作流必须以产品版本 ID 作为业务归属边界。
3. 需求问题、逻辑问题、审批、事件、附件、任务和产物必须归属于具体产品版本。
4. 不同产品版本可处于不同工作流阶段。
5. 同一产品版本同一时间只能有一个活动工作流任务。
6. 不同产品版本可并行存在活动任务，前提是它们之间没有未满足的基线依赖。
7. 当前版本进入工作流后，版本基线不得更换。
8. 现有阶段门禁、回退、终审覆盖和 SDD 重生成能力继续有效。

### FR-11 版本化 PRD 与 SDD

1. 每个 PRD、PRD Review 和 SDD 必须同时标识产品版本和文档修订号。
2. 展示格式应明确区分，例如：
   - 产品版本：`V1.0`
   - PRD 修订：`Revision 2`
3. 同一产品版本内文档修订号独立递增。
4. 不同产品版本的文档修订号互不影响。
5. PRD 应新增“版本目标、基线版本、版本范围、版本差异”内容。
6. SDD 应明确当前版本新增或改变的技术契约，并标记受影响的既有模块。
7. 新版本文档不得覆盖基线版本文档。

### FR-12 版本比较

1. 用户应能比较同一项目的任意两个产品版本。
2. 默认比较当前版本与其基线版本。
3. 比较结果至少包括：
   - 版本目标差异
   - 模块和功能差异
   - 业务规则差异
   - 数据和接口影响
   - 新增、修改、删除统计
   - PRD/SDD 产物状态
4. 比较应优先使用结构化需求对象，不以 Markdown 文本 Diff 作为唯一依据。
5. 用户应能从差异项跳转到对应版本工作区或产物。

### FR-13 项目路线图视图

1. 项目详情首页应从单一工作流页面升级为项目概览。
2. 概览应展示全部版本的顺序、状态、目标、基线、工作流阶段和最新产物。
3. 用户应能按版本状态筛选。
4. 用户应能进入任意产品版本的独立工作区。
5. 路线图必须清晰区分计划版本、进行中版本和已完成版本。
6. 存在版本依赖时，应展示依赖关系和阻塞原因。
7. 项目列表应展示版本数量、当前进行中版本和最近完成版本。

### FR-14 版本操作

1. 用户应能创建、编辑、排序、取消、归档和恢复产品版本。
2. 只有 `PLANNED` 或 `READY` 版本允许修改代码、目标和计划范围。
3. `IN_PROGRESS` 版本的范围变化必须记录变更原因并重新执行差异确认。
4. `COMPLETED` 版本不可修改工作流数据，只能创建后继版本或重新生成其 SDD 修订。
5. 已作为其他版本基线的版本不可取消或删除。
6. 产品版本代码修改后，历史审计和内部关联不得丢失。
7. 版本删除仅允许尚未启动、未被引用且无正式产物的版本。

### FR-15 需求池

1. 项目应提供待规划需求池，用于保存尚未分配到产品版本的需求。
2. 需求池条目应包含稳定 ID、标题、描述、来源、优先级和状态。
3. AI 版本拆分和手工规划都应从同一个需求池选取范围。
4. 需求池条目分配到版本后，应保留其原始来源和分配历史。
5. 从计划版本移出的范围项可返回需求池。
6. 已在完成版本交付的需求不得回到“未规划”状态，只能产生后续变更项。

### FR-16 API 契约

1. API 应提供项目级信息和版本集合的独立资源。
2. 工作流操作路由必须明确包含产品版本标识，避免继续把项目 ID 当工作流 ID。
3. 应支持以下能力：
   - 创建和查询项目
   - 创建、查询、编辑、排序和归档产品版本
   - 生成和确认版本规划建议
   - 创建和确认版本差异
   - 启动版本工作流
   - 查询版本详情、任务、附件和产物
   - 比较两个版本
4. 所有写操作继续使用幂等键。
5. 旧版项目详情和工作流 API 应提供明确迁移期，不得静默改变响应语义。

### FR-17 CLI

1. CLI 应支持创建产品项目。
2. CLI 应支持列出项目版本和查看版本路线。
3. CLI 应支持创建新版本并指定基线。
4. `resume`、`status` 和 `export` 必须要求产品版本上下文。
5. 为降低迁移成本，只有一个版本的项目可暂时允许省略版本参数。
6. CLI 输出必须同时显示项目、产品版本、工作流阶段和文档修订。

### FR-18 迁移与兼容

1. 数据库迁移必须为每个现有项目创建一个默认产品项目和默认产品版本。
2. 默认版本代码建议使用 `V1`，展示名称使用“初始版本”。
3. 现有项目的工作流状态、LLM 配置、需求快照、问题、附件、审批、事件、任务和产物必须迁移到默认版本。
4. 原有项目 ID 应继续可追踪，迁移必须记录新旧对象映射。
5. 迁移不得重新生成或修改现有 PRD/SDD 内容。
6. 迁移完成后，旧项目应在 Web 和 CLI 中可正常访问。
7. 迁移脚本必须支持事务失败回滚和迁移前 SQL 检查。
8. 数据库升级后不得同时存在无法判定归属的孤立工作流数据。

### FR-19 审计与追踪

1. 下列操作必须写入审计事件：
   - 版本创建
   - 基线选择
   - 版本范围确认
   - 差异确认
   - 版本状态变化
   - 版本顺序调整
   - 版本取消、归档和恢复
2. 审计事件必须包含项目 ID、产品版本 ID、操作类型、时间和关键详情。
3. AI 建议与人工确认必须分开记录。
4. 用户应能从版本详情查看版本相关事件。

## User Stories

1. As a 产品负责人, I want 在一个项目下规划 MVP、V1 和 V2, so that 我可以管理长期产品迭代。
2. As a 产品负责人, I want 为每个版本定义独立目标, so that 团队知道该版本为什么存在。
3. As a 产品负责人, I want 明确每个版本的范围和非目标, so that 版本不会持续膨胀。
4. As a 产品负责人, I want 将候选需求放入统一需求池, so that 尚未排期的需求不会丢失。
5. As a 产品负责人, I want 让 Agent 建议版本拆分, so that 我可以快速形成首版路线。
6. As a 产品负责人, I want 审核和修改 Agent 的版本建议, so that 最终规划仍由我决定。
7. As a 产品负责人, I want 调整未启动版本的顺序, so that 路线可以响应业务优先级变化。
8. As a 产品负责人, I want 从上一完成版本创建新版本, so that 不必重新描述所有已有能力。
9. As a 产品负责人, I want 选择非最近版本作为基线, so that 可以支持特殊迭代分支。
10. As a 产品负责人, I want 查看继承了哪些需求, so that 我知道新版本的完整产品基线。
11. As a 产品负责人, I want 输入本版本增量需求, so that 系统只重点分析变化部分。
12. As a 产品负责人, I want 查看新增、修改、删除和不变项, so that 版本差异清晰可控。
13. As a 产品负责人, I want 逐项确认修改和删除, so that Agent 不会擅自破坏已有需求。
14. As a 产品负责人, I want 看到删除功能的影响分析, so that 我可以评估兼容性风险。
15. As a 产品负责人, I want 在范围确认前看到门禁错误, so that 不完整版本不会启动工作流。
16. As a 产品负责人, I want 每个版本独立运行需求工作流, so that 不同迭代的状态和文档不会混淆。
17. As a 产品负责人, I want 查看项目路线图, so that 我可以掌握全部版本的进展。
18. As a 产品负责人, I want 比较任意两个版本, so that 我可以解释产品发生了什么变化。
19. As a 产品负责人, I want 归档旧版本但保留记录, so that 项目界面保持简洁。
20. As a 产品负责人, I want 取消尚未开始的版本, so that 已放弃的规划不会继续占用路线。
21. As a 需求分析人员, I want 使用稳定需求 ID 跟踪跨版本变化, so that 同一能力不会被误认为多个需求。
22. As a 需求分析人员, I want 识别重复或冲突的版本范围, so that 不同版本不会重复承诺同一首发能力。
23. As a 需求分析人员, I want 查看版本范围的来源, so that 每项需求都可追溯。
24. As a 技术负责人, I want 查看当前版本相对基线的技术影响, so that 可以评估改造范围。
25. As a 技术负责人, I want 在 SDD 中区分新增和受影响模块, so that 实现工作可以围绕增量组织。
26. As a 技术负责人, I want 查看版本依赖和阻塞关系, so that 前置版本未完成时不会错误推进。
27. As a 用户, I want 明确区分产品版本与 PRD 修订号, so that 不会把 PRD v2 误认为产品 V2。
28. As a 用户, I want 在项目列表看到当前版本, so that 我可以快速进入正在推进的迭代。
29. As a 用户, I want 从差异项跳转到具体文档, so that 我可以核对变化的详细定义。
30. As a CLI 用户, I want 指定产品版本执行 status 和 export, so that 命令行结果不会产生歧义。
31. As a 老用户, I want 升级后继续看到原来的项目和文档, so that 新模型不会破坏已有数据。
32. As a 老用户, I want 旧项目自动获得一个初始版本, so that 我无需手工整理迁移数据。
33. As a 系统, I want 将所有工作流数据绑定到产品版本, so that 多版本数据不会串联。
34. As a 系统, I want 阻止循环版本依赖, so that 版本路线始终可执行。
35. As a 系统, I want 阻止删除已被后续版本引用的基线, so that 继承链不会断裂。
36. As a 系统, I want 保留 AI 建议和人工确认差异, so that 规划过程可以审计。
37. As a 系统管理员, I want 使用数据库迁移完成模型升级, so that 现有安装可以平滑进入 v0.2。

## Implementation Decisions

- 新增 `Project Version` 作为工作流聚合根；`Project` 退回长期产品容器职责。
- 不复用现有产物 `version` 字段表示产品版本。产品版本使用独立 ID 和 `version_code`，产物继续使用修订号。
- 产品版本状态与现有九阶段工作流阶段分离，避免规划状态和文档生成状态互相污染。
- 每个工作流相关记录必须直接或可无歧义地关联产品版本。
- 基线继承采用“创建独立快照”的方式，不对上一版本需求对象建立可变共享引用。
- 跨版本追踪依赖稳定需求 ID；新版本继承时保留稳定 ID，新增需求分配新 ID。
- 版本差异以结构化对象比较为主，Markdown 文本比较仅作为辅助展示。
- `MODIFIED` 和 `REMOVED` 必须人工确认；LLM 不拥有破坏性变更决策权。
- AI 版本拆分是建议流程，不直接改变正式版本路线。
- 版本规划优先采用纵向用户价值切片，避免按前端、后端、数据库等技术层横向拆版本。
- 项目级信息在版本启动时形成快照，保证历史版本可复现。
- 同一产品版本保持任务互斥；不同产品版本允许独立排队和执行。
- v0.1 迁移采用“一项目一默认版本”兼容方案，不修改已有产物正文。
- API 路径和返回对象逐步引入版本上下文，旧 API 只在明确兼容期内保留。
- Web 信息架构调整为“项目概览 → 版本工作区 → 产物详情”三级结构。
- 本版本不新增技术专家评审和技术闭环阶段，现有九阶段工作流保持不变。

## API 设计原则

- 项目资源表示长期产品，不再直接返回单一工作流状态。
- 产品版本资源返回版本计划状态和当前工作流摘要。
- 工作流写操作统一在产品版本上下文执行。
- 项目概览接口应一次返回轻量版本摘要，避免为每个版本逐一发起请求。
- 版本详情接口返回范围、差异、门禁、工作流、附件和产物摘要。
- 大型版本比较结果可以异步生成，但必须支持幂等和状态查询。
- 所有错误应明确指出是项目级、版本规划级还是工作流级错误。

## Testing Decisions

### 测试原则

- 测试用户可观察的项目、产品版本和工作流行为，不测试私有函数实现。
- 最高测试接缝优先使用“创建项目 → 规划版本 → 确认差异 → 完成工作流”的完整业务路径。
- 迁移测试必须使用接近真实 v0.1 数据的数据库快照。
- 并发测试必须验证不同版本可并行、同一版本仍互斥。
- 所有破坏性版本操作都必须有拒绝路径测试。

### 模型与规则测试

- 同项目版本代码唯一。
- 显式排序不依赖版本代码。
- 基线必须属于同项目。
- 版本依赖不得成环。
- 被引用基线不得删除。
- 产品版本状态转换合法。
- 产品版本号与产物修订号互相独立。

### 版本规划测试

- AI 建议不会未经确认写入正式路线。
- 用户可部分采纳版本建议。
- 需求池条目只能有一个首发版本。
- 未启动版本可移动范围项。
- 已启动版本变更触发重新确认。

### 继承与差异测试

- 基线最终需求被完整复制。
- 继承不会修改基线记录。
- 稳定需求 ID 跨版本保留。
- 新增需求获得新 ID。
- 新增、修改、删除和不变项识别正确。
- 修改和删除未确认时门禁拒绝启动。
- 删除项包含影响分析。

### 工作流隔离测试

- 两个产品版本的问题、任务、附件和产物完全隔离。
- 同一版本重复写请求保持幂等。
- 同一版本拒绝并发活动任务。
- 不同版本可分别排队执行。
- 版本工作流完成后自动更新版本状态。
- 回退只影响当前产品版本。

### API 与 Web 测试

- 项目概览返回全部轻量版本摘要。
- 产品版本创建、编辑、排序、归档和比较请求正确。
- 工作流路由包含正确版本上下文。
- 项目列表展示当前版本。
- 项目概览与版本工作区路由正确。
- 页面明确区分产品版本和文档修订。
- 版本门禁错误、空状态、加载状态和失败状态完整。

### 迁移测试

- 每个 v0.1 项目产生一个默认产品版本。
- 旧工作流状态和所有子记录迁移完整。
- 原产物正文和修订号不变。
- 新旧对象映射可追踪。
- 迁移失败时事务完整回滚。
- 迁移后不存在孤立记录。
- 迁移后的旧项目可继续执行或查看。

### 核心验收场景

1. 用户创建一个产品项目，并由 AI 生成 MVP、V1、V2 三个建议版本。
2. 用户修改建议并确认版本路线。
3. 用户完成 MVP 的九阶段工作流。
4. 用户以 MVP 为基线创建 V1，输入增量需求。
5. 系统识别新增、修改和删除项，用户逐项确认。
6. V1 启动独立工作流并生成自己的 PRD Revision 1 和 SDD Revision 1。
7. 用户比较 MVP 与 V1，能够看到结构化差异。
8. MVP 的历史需求和产物保持不变。

## 交付分期

### Phase 1：模型分层与兼容迁移

- Project 与 Project Version 分层
- 工作流数据归属产品版本
- v0.1 项目自动迁移
- 项目概览与版本工作区基础路由

完成标准：旧项目无损迁移，新建项目可包含多个独立版本。

### Phase 2：手工版本规划

- 版本路线
- 需求池
- 版本范围
- 基线选择
- 版本状态和范围确认门禁

完成标准：不依赖 AI 也能完整规划并启动多个产品版本。

### Phase 3：继承、差异与比较

- 基线需求继承
- Version Change Set
- 破坏性变更确认
- 版本比较页面
- PRD/SDD 版本上下文增强

完成标准：用户能基于上一版本以增量方式创建并完成新版本。

### Phase 4：AI 辅助版本规划

- 版本规划 Agent
- AI 建议预览
- 部分采纳和人工调整
- 建议与确认审计

完成标准：Agent 能提出可编辑的纵向版本切片，且无法绕过人工确认。

## Out of Scope

1. 自动研发排期、工时估算和人员分配。
2. Sprint、看板、燃尽图和研发任务管理。
3. Git 分支、Commit、Pull Request 和代码版本关联。
4. 自动部署、灰度发布、回滚和线上版本管理。
5. 应用商店版本、安装包或制品仓库管理。
6. 多项目之间的版本依赖。
7. 同一产品版本的多分支合并和冲突解决。
8. 多用户协同、RBAC、审批人分配和消息通知。
9. 技术栈前置录入、技术专家评审和 SDD 后技术闭环。
10. 自动生成实现代码。
11. 将 PRD/SDD 文档修订号自动映射为产品版本号。
12. 复杂的发布日期预测和资源容量规划。

## Risks

| 风险 | 应对策略 |
| --- | --- |
| Project 语义改变导致大范围迁移 | 先建立兼容层和迁移测试，再调整 API 与 Web |
| 版本继承产生隐式共享修改 | 继承时创建不可回写基线的独立快照 |
| 稳定需求 ID 难以准确匹配 | 继承保留 ID，新增项由系统分配，模糊匹配必须人工确认 |
| AI 将技术模块误当产品版本 | Prompt 和 Guardrail 强制纵向用户价值切片 |
| 产品版本与文档版本继续混淆 | 数据字段、API、界面统一使用 Product Version 与 Revision 文案 |
| 多版本并行引入任务竞争 | 任务互斥键从项目级调整为产品版本级 |
| 旧 API 调用方中断 | 提供迁移期、兼容响应和明确弃用提示 |
| 版本范围频繁变化绕过门禁 | 已启动版本变更必须形成 Change Set 并重新确认 |

## Further Notes

- v0.2 的核心价值不是增加一个版本号字段，而是把“项目容器”和“工作流实例”拆开。
- 当前九阶段工作流可以复用，但所有状态和子数据的归属边界必须从项目改为产品版本。
- 产品版本规划应先支持完全手工操作，再叠加 AI 建议，避免核心能力依赖模型质量。
- 版本继承以最终确认的结构化需求为事实基线，不以旧 PRD Markdown 作为唯一输入。
- 技术专家评审和技术闭环仍然重要，但属于独立版本能力，不能隐藏在本次版本规划改造中。
