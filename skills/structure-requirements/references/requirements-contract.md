# 结构化需求契约

## 必要信息

每个模块记录名称、职责及功能点。每个功能点至少说明：

| 字段 | 含义 |
|---|---|
| name | 功能名称 |
| description | 用户价值和行为范围 |
| dataSource | 数据来自哪里 |
| interactionLogic | 用户触发、系统处理和反馈 |
| operationImpact | 对数据、用户和其他模块的影响 |
| dependencies | 前置能力和下游影响 |

另外全局记录：

- 数据来源：类型、owner、更新时效。
- 交互：触发条件、系统行为、用户反馈。
- 操作影响：生效时间、影响对象、是否可恢复。
- 依赖：上下游、先后顺序、删除影响。
- 状态、规则、异常、假设。

## 问题质量

- 一条问题只询问一个决策。
- 问题必须能由用户直接回答。
- 问题类型可使用模块、数据来源、交互、操作影响、依赖、边界或异常。
- `importance` 解释不回答会影响哪个决策。
- 已回答问题保留原编号并标记 `resolved`，不要删除审计痕迹。

## 中文展示

使用“第 N 轮需求理解”标题，包含完整理解表、仍需澄清问题表和本轮数量。
展示文本不得代替结构化字段。
