# SDD生成

## 基本原则

1. 严格基于终审通过的PRD，不补充新的业务能力。
2. 每个业务模块生成一份SDD。
3. 每份SDD严格使用八个固定章节，顺序不可改变。
4. 请求模型、响应模型和数据库实体分离。
5. 技术默认值必须显式列出，不得伪装成PRD事实。
6. 不适用的页面、API或数据库章节写明不适用原因，不臆造实现。
7. 信息不足时写 `【待补充：PRD未提供xxx信息】`，继续其他部分。

阅读 [references/sdd-contract.md](references/sdd-contract.md) 获取完整章节契约。
