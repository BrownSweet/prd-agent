# SDD八章契约

每个模块按以下顺序使用二级标题。

## 1. 功能契约

- 目标不超过30字。
- 范围不超过5条，均以动词开头。
- 至少列出2条不包含功能。
- 定义性能、安全、可用性、日志监控。
- 列出前端、后端、数据库、中间件和必需环境变量。
- PRD未指定技术栈时默认 React 18 + TypeScript + Tailwind CSS、
  Go 1.21+ + Gin + GORM、MySQL 8.0+，并记录为技术默认值。

## 2. 页面结构与组件树

- 使用ASCII树，标注路由。
- 每个组件定义Props、State、Events。
- 页面覆盖idle、loading、empty、error、success。
- 无页面模块写明不适用，不虚构页面。

## 3. 数据模型定义

- 前端使用带注释的TypeScript `interface`，camelCase。
- Go使用带json tag和binding校验的struct；Python使用Pydantic Field。
- 请求、响应和数据库实体禁止混用。

## 4. API接口定义

每个接口包含路径、方法、认证、路径参数、请求JSON、统一成功响应和至少两种
统一错误响应。路径统一使用 `/api/v1` 前缀。明确参数校验、业务校验、幂等、
事务边界、外部依赖超时与重试、成功失败后的状态变化。

## 5. 数据库表设计

- 表格列出字段名、类型、约束、说明。
- 包含主键、索引、默认值、可空性、created_at、updated_at。
- 提供完整DDL和索引。
- 说明迁移、事务和缓存一致性。
- 无持久化需求时写明不适用。

## 6. 核心交互流程

编号描述用户操作、系统响应、状态变化、成功失败分支，以及网络、超时、空数据
和服务错误。标注 `idle → loading → success/error` 等状态流转与最终状态。

## 7. 测试验收标准

使用表格：用例ID、场景、前置条件、操作、预期结果、是否可自动化。至少覆盖
2个正常场景、2个异常场景和2个边界条件。

## 8. 技术决策与约束

- TypeScript禁止`any`，Go禁止`interface{}`。
- 错误必须处理，状态不得未定义。
- 前端camelCase、后端字段snake_case、API路径kebab-case。
- 函数单一职责且不超过50行。
- 禁止硬编码密钥和URL、禁止吞错、禁止未经审计的依赖。
- 指出未来变更点以及接口抽象、依赖注入或策略模式等扩展方式。
